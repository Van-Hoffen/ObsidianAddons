# README — Obsidian QuickAdd Obfuscation Script

Скрипт для Obsidian + QuickAdd, который позволяет по горячей клавише:
- обфусцировать выделенный текст по набору regex-правил;
- сохранять карту замен в конце заметки;
- восстанавливать исходный текст обратно по токенам. [cite:3][cite:6]

Подходит для сокрытия email, URL, доменов и любых пользовательских шаблонов текста, которые удобно описать регулярными выражениями. [cite:3][cite:6]

## Что делает скрипт

Скрипт берёт **выделенный текст** в активной markdown-заметке, прогоняет его по списку правил и заменяет найденные фрагменты на токены вида `[[mail_1]]`, `[[url_1]]`, `[[org_1]]`. [cite:6]

Для каждой замены он сохраняет исходное значение в JSON-карту, которая записывается в конец заметки внутри блока `%% ... %%`; такой блок в Obsidian используется как комментарий и не показывается в reading mode. [cite:19][cite:24]

## Как это работает

Скрипт экспортирует две функции для QuickAdd:
- `obfuscateSelection`
- `restoreSelection` [cite:6]

Их можно привязать к двум разным Macro choice в QuickAdd, а затем назначить на горячие клавиши через `Settings → Hotkeys`. [cite:5][cite:6][cite:8]

## Установка

1. Установить community plugin **QuickAdd** в Obsidian. [cite:15][cite:12]
2. Положить файл `obfuscate.js` в папку vault, например `scripts/obfuscate.js`; QuickAdd использует user scripts из обычных `.js` файлов внутри vault. [cite:6][cite:17]
3. Открыть `QuickAdd → Manage Macros` и создать макрос `Obfuscate selection`. [cite:5][cite:6]
4. Добавить в него шаг **User Script** и выбрать export `obfuscateSelection`. [cite:6]
5. Аналогично создать второй макрос `Restore selection` и выбрать export `restoreSelection`. [cite:6]
6. Назначить горячие клавиши в `Settings → Hotkeys`. [cite:8]

## Использование

### Обфускация

1. Открыть заметку.
2. Выделить нужный фрагмент текста.
3. Нажать хоткей команды `Obfuscate selection`.

Скрипт заменит найденные значения на токены и добавит карту замен в конец статьи. [cite:6][cite:19]

### Восстановление

1. Выделить текст с токенами.
2. Нажать хоткей команды `Restore selection`.

Скрипт найдёт токены в выделении, посмотрит их в карте замен и вернёт исходные значения. [cite:6]

## Формат карты замен

В конце заметки создаётся служебный блок такого вида:

```md
%%obfuscation-map-start%%
{
  "mail_1": {
    "original": "user@example.com",
    "replacement": "user@example.com",
    "prefix": "mail"
  },
  "url_1": {
    "original": "https://example.com/path",
    "replacement": "https://example.com/path",
    "prefix": "url"
  }
}
%%obfuscation-map-end%%
```

Этот блок нужен для обратного восстановления текста. [cite:19][cite:24]

## Правила замены

В скрипте есть два массива:
- `DEFAULT_RULES` — стандартные правила;
- `CUSTOM_RULES` — пользовательские правила. 

Они объединяются в общий список `RULES`, который выполняется сверху вниз. [cite:3][cite:6]

### Структура одного правила

```js
{
  prefix: "org",
  regex: /\bplanet(?=\W|$)/gi,
  replaceWith: "jupiter",
  storeOriginal: true
}
```

### Поля

- `prefix` — префикс токена, например `org_1`, `mail_2`.
- `regex` — JavaScript RegExp; желательно всегда использовать флаг `g`.
- `replaceWith` — строка замены; если `null`, будет создан токен без прямой текстовой подмены.
- `storeOriginal` — если `true`, исходное значение попадёт в карту замен. [cite:3][cite:6]

## Почему порядок правил важен

Regex-правила выполняются последовательно, и более общее правило может перехватить текст раньше более точного. Поэтому email и URL нужно обрабатывать **до** доменов, иначе домен внутри email или URL будет совпадать отдельно. [cite:3][cite:6]

Пример правильного порядка:
- URL
- email
- domain
- пользовательские точечные правила [cite:3][cite:6]

Если у тебя есть одновременно правило на `planet.ru` и отдельное правило на `.ru`, правило на `planet.ru` должно стоять выше. [cite:3]

## Предупреждения по regex

- Некоторые regex могут **пересекаться** друг с другом; в таком случае сработает первое совпавшее правило. [cite:3]
- Если использовать `\W` прямо в match, этот символ тоже попадёт в совпадение; часто безопаснее использовать lookahead, например `(?=\W|$)`. [cite:3]
- Без флага `g` regex обычно заменяет только первое совпадение. [cite:3]
- Если одно правило уже заменило текст на токен, следующие правила этот фрагмент больше не увидят. [cite:6]

## Примеры пользовательских правил

### Замена слова

```js
{
  prefix: "org",
  regex: /\b[pP]lanet(?=\W|$)/g,
  replaceWith: "jupiter",
  storeOriginal: true
}
```

### Замена доменной зоны

```js
{
  prefix: "zone",
  regex: /\.[rR][uU]\b/g,
  replaceWith: ".com",
  storeOriginal: true
}
```

### Замена города

```js
{
  prefix: "city",
  regex: /\b[Mm]oscow\b/g,
  replaceWith: "London",
  storeOriginal: true
}
```

### Простая замена без записи в карту

```js
{
  prefix: "brand",
  regex: /\bGoogle\b/g,
  replaceWith: "Goog",
  storeOriginal: false
}
```

## Пример результата

Исходный текст:

```md
Напиши на user@example.com или зайди на https://example.com
```

После обфускации:

```md
Напиши на [[mail_1]] или зайди на [[url_1]]
```

Внизу заметки появится карта замен, по которой можно сделать обратное восстановление. [cite:6][cite:19]

## Ограничения

Скрипт работает с **выделенным текстом**, а не со всей заметкой целиком. Это удобно для точечной обработки, но важно помнить, что невыделенный текст не будет изменён. [cite:6]

QuickAdd запускает этот код как user script внутри Obsidian, поэтому файл должен лежать в vault как обычный `.js` файл. [cite:6][cite:17]

## Что можно улучшить дальше

- сделать одну toggle-команду вместо двух отдельных;
- вынести `CUSTOM_RULES` в отдельный JSON-файл;
- добавить поддержку групп `$1`, `$2` в `replaceWith`;
- добавить обработку markdown-ссылок вида `[текст](url)`. [cite:3][cite:6]
